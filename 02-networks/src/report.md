## Part 3. Утилита iperf3
3.1 Скорость соединения
1. 8 Mbps в MB/s.  
8 Mbps = 8 Мегабит в секунду.  
1 Байт = 8 бит.  
8 Mbps = 8 / 8 = 1 MB/s

2. 100 MB/s в Kbps.  
100 MB/s = 100 Мегабайт в секунду.  
1 Мегабайт = 1024 Килобайт = 8192 Килобит.  
100 MB/s = 100 * 8192 = 819,200 Kbps

3. 1 Gbps в Mbps.  
1 Gbps = 1000 Mbps

## Part 4. Сетевой экран

4.1. Утилита iptables
Содержанием файла /etc/firewall 
![Интерфейсы ws1,ws2 ](pictures/4-1_firewallws1ws2.png)

1. Создали файл скрипта командой `sudo nano /etc/firewall.sh`
2. Запуск `sudo /etc/firewall.sh`
3. Просмотр правил `sudo iptables -L -n -v (-l list, -n ip` адреса и порты в числовом формате, -v подробный отчет)
4.  Суть стратегий:
На машине WS1:
Применена стратегия «сначала запрет, потом разрешение».
В начале цепочки OUTPUT находится правило `DROP icmp --icmp-type echo-reply`
После него следует правило `ACCEPT icmp --icmp-type echo-reply`
На машине WS2:
Применена стратегия «сначала разрешение, потом запрет».
В начале цепочки OUTPUT находится правило `ACCEPT icmp --icmp-type echo-reply`
После него следует правило `DROP icmp --icmp-type echo-reply`
5. iptables обрабатывает правила последовательно сверху вниз. Если правило подходит — применяется указанное действие (ACCEPT/DROP) и дальнейшая проверка прекращается
6. Для WS1 (сначала запрет):
- Пакеты ICMP echo-reply (ответы на ping) находят совпадение с первым правилом DROP
- Действие DROP применяется немедленно 
- Второе правило ACCEPT никогда не достигается
- Результат: машина WS1 не отвечает на ping (100% потерь пакетов)

Для WS2 (сначала разрешение):
- Пакеты ICMP echo-reply находят совпадение с первым правилом ACCEPT
- Действие ACCEPT применяется немедленно
- Второе правило DROP никогда не достигается
- Результат: машина WS2 успешно отвечает на ping (0% потерь пакетов)

Вопрос: Почему для адреса 10.10.0.0/18 был выбран маршрут, отличный от 0.0.0.0/0, хотя он попадает под маршрут по умолчанию?

Ответ:
Маски подсети:  
/18 = 255.255.192.0 = первые 18 бит фиксированы  
/0 = 0.0.0.0 = нет фиксированных битов (подходит под всё)

Принцип Longest Prefix Match (наиболее специфичный маршрут): Когда ядро ищет маршрут для пакета, оно выбирает маршрут с самой длинной маской (наиболее специфичный).
Для пакетов в сеть 10.10.0.0/18 ядро выберет более специфичный маршрут (/18), а не общий default route (/0)

пример:
Представь две подсказки:  
"Это число от 1 до 1000" (очень общая)  
"Это число от 200 до 250" (более конкретная)  
Если правильный ответ — 225, какая подсказка лучше?

Обе подсказки верны (225 попадает в обе категории)  
Но вторая подсказка точнее, поэтому используем ее.


5.5. Построение списка маршрутизаторов  
на r1 `sudo tcpdump -tnv -i enp0s8 icmp`  
Флаги:
- t — не показывать timestamp
- n — не преобразовывать адреса в имена
- v — подробный вывод
- i enp0s8 — интерфейс к WS11

icmp — только ICMP пакеты (которые использует traceroute)

вывод терминалов:
![Интерфейс ws11+r1](pictures/5-5-tracepath.png)

Вопрос: принцип работы построения пути при помощи traceroute.

Ответ: 
Вывод из tcpdump:

Первый пакет (TTL=1) дошёл только до R1 (10.10.0.1), который отправил ICMP "Time exceeded"

Второй пакет (TTL=1) подтвердил первый хоп

Третий пакет (TTL=2) прошёл через R1 и дошёл до R2 (10.100.0.12)

Четвёртый пакет (TTL=3) прошёл через R1 и R2, достиг WS21 (10.20.0.10)

Принцип работы traceroute: Утилита отправляет пакеты с увеличивающимся TTL. Каждый роутер уменьшает TTL на 1. Когда TTL достигает 0, роутер отбрасывает пакет и отправляет ICMP-ответ "Time exceeded". Анализируя эти ответы, traceroute строит путь до целевого хоста.

5.6. Использование протокола ICMP при маршрутизации

ws11 `ping -c 1 10.30.0.111` и вывод с r1 `sudo tcpdump -n -i enp0s8 icmp`
![Интерфейс ws11+r1](pictures/5-6-ping.png)

При ping несуществующего IP 10.30.0.111 видно, что пакет доходит до первого роутера (R1), но R1 не отправляет ICMP "Destination Net Unreachable". Это связано с тем, что у R1 есть маршрут по умолчанию, поэтому он пытается переслать пакет дальше, а не сообщать об ошибке. Пакет теряется где-то дальше в сети. При пинге с ws11 видно, что пакеты потерялись.

![Интерфейc ws11](pictures/8-1-ws11_check.png) 

Remote Forwarding (ws11-r2-ws22): Так как Apache на ws22 слушает только localhost:80, Remote Forwarding запускается с самой ws22 на r2 (порт 8081), откуда доступен с ws11.
